## 1. Executive Summary

AutoService 是 OneSync 旗下 **OaaS (Organization-as-a-Service)** 框架下的第一个落地产品 —— 一个 7×24 的 AI 客服组织。我们为**中小商户**提供一套"AI 团队 + 人工 Copilot + 闲时自学习"的客服系统,让商户能在 **2 小时内自助上线**,把 80% 的重复咨询交给 AI,把人留给关键时刻。相较传统客服 SaaS,AutoService 的差异点是:**快慢双模型**消除"AI 慢"焦虑、**Copilot 默认**模式让人机协作像真实团队一样工作、**Dream Engine 闲时整理记忆**让系统每天醒来都比昨天聪明一点。目标是在稳态运行下把事实准确率维持在 92%,把首次响应压到 60 秒以内,把升级转结案率推到 89% 以上。

## 2. Problem Statement

### Who has this problem?
- **中小商户**(1–200 人规模):电商、SaaS、本地服务、跨境贸易等,有线上咨询场景但养不起 24 小时客服团队
- **商户的人工客服**:3–20 人小团队,同时要回答重复问题和处理复杂投诉,疲于切换
- **商户的 C 端客户**:跨时区、多语言、带历史订单上下文,对"等待"和"被机器人糊弄"非常敏感

### What is the problem?
现有客服方案是"二选一"困境:
1. **纯人工客服** → 成本高 / 非工作时间无人 / 高峰期排队 → 客户流失
2. **传统 AI 客服(规则机器人 / 早期 LLM bot)** → 回答僵硬、答非所问、幻觉严重、转人工体验割裂 → 客户反感
3. **试图自己接 LLM API** → 需 2–8 周工程投入、无合规、无多模型编排、无人机协作 → 大部分商户放弃

### Why is it painful?
- **商户**: 咨询高峰接不住 → 转化率损失;半夜无人值守 → 国际客户流失;雇 AI 公司做定制 → 慢且贵
- **客服**: 80% 时间在回答重复问题,失去处理真正复杂问题的精力和满足感
- **C 端客户**: 等待 + 幻觉 + 转人工上下文丢失 = 信任崩塌

### Evidence
- 某电信运营服务商实测数据:稳态下 AI 可消化 **80%** 重复咨询,事实准确率 **92%**(来源于 Slide 8 真实租户数据)
- Slide 9 SLA 表反映行业通病:人工接单等待是"沉默危险区",超过 180s 客户流失率陡增
- 反复出现的商户诉求关键词: "2 小时上线"、"不用请工程师"、"沙箱先试"

## 3. Target Users & Personas

### Primary · 商户管理员 "老陈"(采购 & 配置决策人)
- **身份**: 中小电商老板 / SaaS COO
- **目标**: 尽快让客服系统跑起来,不想招人、不想等工程排期
- **痛点**: 看不懂 AI 技术栈 / 担心幻觉 / 担心合规
- **行为**: 注册 → 上传资料 → 沙箱试一遍 → 拉团队进来 → 对外开放
- **决策标准**: 能否在一个下午搞定、出问题能不能回滚

### Primary · 人工客服 "小李"(日常使用者)
- **身份**: 3–10 人客服团队成员
- **目标**: 不被 AI 取代,和 AI 组成"小分队"提高自己的产出
- **痛点**: 现有工具让她要在多个 chat 窗口之间来回切 / 看不到 AI 给客户说了什么 / 接手时上下文丢失
- **行为**: 打开 IM 工作区 → 盯"Agent 分队"实时刷新的卡片 → 点开时进入 Copilot → 需要时 `/hijack`
- **决策标准**: 顺手、不打扰、能看到 AI 在干什么

### Primary · C 端客户 "David"(最终体验者)
- **身份**: 普通购物者 / 咨询者,多语言、多时区
- **目标**: 问题被解决,不被推诿
- **痛点**: 等得久 / 被机器人循环 / 转人工要重新说一遍
- **行为**: 点独立站浮动按钮 → 问问题 → 要么得到答案,要么顺滑升级到人工
- **决策标准**: 快、准、不被糊弄

### Secondary · 平台运营(OneSync 自己)
- **身份**: 平台 SRE / 合规 / 计费团队
- **目标**: 多租户 SLA 达标、合规无事故、按指标计费
- **行为**: 维护 SLA 监控、合规规则库、灰度通道、计费引擎

### Jobs-to-be-done
- 商户: "当我的店半夜有国际客户来问问题时,我想要 AI 用客户母语准确回答,这样我就不会错过订单"
- 客服: "当同时有 5 个客户在问时,我想让 AI 先应付简单的,这样我能专注处理最棘手的那一个"
- 客户: "当我问一个复杂问题时,我想要系统先告诉我'在查了',这样我不会觉得被晾着"

## 4. Strategic Context

### 业务目标
- 成为 **OneSync OaaS 框架的首个旗舰产品**,验证 Socialware "组织即服务"的商业化路径
- 在 2026 H2 实现 100 家付费商户、稳态 CSAT ≥ 4.5

### 市场机会
- 全球客服 SaaS 市场超 300 亿美元,AI 客服子赛道年增速 > 30%
- 中小商户是被"Zendesk + 人工外包"夹击的空白地带 —— 要么太贵太重,要么太弱

### 为什么是现在
- 大模型成本 2024–2026 下降一个数量级,快慢双模型架构在经济上首次可行
- 中小商户对 AI 的接受度在 2025 年后显著转正,愿意为"2 小时上线"付费
- 多语言 LLM 质量跨过可商用阈值,一个 Agent 覆盖 20+ 种语言成为可能

### 竞争格局与差异化

| 上线时间 | 2–8 周 | 数天 | **2 小时自助** |
| 人机协作 | 割裂转接 | 无 | **Copilot 默认 + 角色翻转** |
| 事实准确率 | N/A | 60–80% | **92% 稳态** |
| 自学习 | 无 | 无 | **闲时整理记忆 + 灰度** |
| 合规 | 手动 | 无 | **按地区自动预检** |

## 5. Solution Overview

### 产品构成(4 个核心模块)

**模块 A · 多触点接入层**
独立站浮动按钮 / IM / PSTN 电话呼入,统一由 Agent 接管,客户从任何渠道进来体验一致。

**模块 B · AI 客服团队(4 个专精角色)**
- 客服 Agent(知识库问答 · 反幻觉 · 多轮记忆 · 情绪识别)
- 翻译 Agent(20+ 语言 · 自动检测 · 术语映射 · 双向流式)
- 线索收集 Agent(四要素提取 · 意向评估 · CRM 直通)
- 智能分流 Agent(信心评估 · 对话摘要 · 团队匹配 · SLA 守护)

**模块 C · 人机协作工作区**
- 每位人工客服拥有"Agent 分队"频道(= 客服小分队),N 个 Agent 向其汇报
- Agent **接洽**客户后在分队频道发**实时刷新卡片**(不点开时只看摘要)
- **对话监管**: 点开卡片弹出聊天窗,默认 Copilot 模式 (Agent driver)
- **人工建议**: 人工在窗内提议,Agent 采纳后发送给客户
- **人工提醒**: (a) Agent `@人工` 求助 / (b) 人工 `/hijack` 抢单
- **角色翻转**: 人工 driver,Agent 退到侧栏继续副驾驶

**模块 D · Dream Engine 闲时学习循环**
商户先在管理群和 Dream Engine 对话**配置学习规则**(触发时机 / 覆盖范围 / 风险阈值 / 灰度策略) → 白天所有对话写临时记忆池 → 业务低峰 Dream Engine 自动启动 → 回放整理 → 沉淀结构化经验 → 晨起推送提案到管理群 → 商户管理员审核 → 5%→100% 灰度 → `/rollback` 可撤回。平台的合规规则通过下发的"闲时学习策略"模板预置,商户配置叠加其上。

### 核心用户流程(三视角)

详见已可视化的两个 widget:
1. **对话级流程**: 6 步状态机(顾客触发 → Agent 接洽 → 对话监管 → 人工建议 → 人工提醒 → 角色翻转),C 端 / 商户 / 平台 三泳道并行
2. **商户生命周期**: 阶段 A 自助上线(≤2h) → 阶段 B 日常运营(实时循环) → 阶段 C 闲时学习(夜间循环)

### 关键设计决策(已和用户确认)
1. Copilot 模式对**所有对话**开放,点开卡片进入对话监管,或 `@agent` 主动介入;不点开时卡片只做实时摘要刷新
2. 一个人工 : N 个 Agent,具体分配规则在**管理群**里动态定义("Agent 分队"= 客服小分队)
3. **接管次数**(计费指标)= 角色翻转次数,**提议次数**作为另一独立指标记录
4. **闲时学习**由商户管理员拍板;平台合规预审只通过**策略模板下发**实现
5. 客户进来系统默认加载历史上下文;新客户自动创建工作目录用于未来收集

## 6. Success Metrics

### 计费指标(★,产品价值承诺)
| 指标 | 定义 | 稳态目标 |
|---|---|---|
| ★ 接管次数 | 角色翻转完成次数/月 | 阶梯计费 |
| ★ 客户满意度 CSAT | 结案后评分均值 | ≥ 4.5 / 5.0 |
| ★ 升级转结案率 | 翻转后成功结案 / 翻转总数 | ≥ 89% |

### 运营指标(监控用,不计费)
- 事实准确率: ≥ 92%(稳态) · 平均首回: < 60s · 接单等待: < 180s · 平均会话时长: 5–15 分钟 · AI 提议采纳率 · AI 消化率(应 ≥ 80%)

### SLA(平台对商户的承诺)
- onboard 首屏应答 < 3s · 占位消息 < 1s · 慢查询续写 < 15s · 人工接单等待 < 180s · 人工首次回复 < 60s · 自助上线 < 2h

### 北极星指标
**每月升级转结案总数 × CSAT** —— 同时反映规模、质量、人机协作效率

## 7. User Stories & Requirements

完整 user story 清单见 `AutoService-UserStories.md`,按四个 Epic 组织:

- **Epic 1 · 自助上线**: 商户 2h 内从注册到沙箱可用(4 个 story)
- **Epic 2 · 实时对话**: C 端 → AI → 人工 的 6 步无缝体验(6 个 story)
- **Epic 3 · 双账本与仪表盘**: 商户看得到 AI 和人工的协作效率(3 个 story)
- **Epic 4 · 闲时学习**: 商户配置规则 + Dream Engine 每晚自学习 + 灰度,次日更聪明(4 个 story)

### Epic 假设(Hypothesis)
> **我们相信**: 如果中小商户能在 2 小时内自助上线一个 Copilot 默认 + Dream Engine 闲时自学习的 AI 客服团队,**那么**他们会获得 80% 重复咨询消化 + 92% 事实准确率 + 89% 升级结案率的稳态体验,**我们将看到** 付费留存 ≥ 70% 和 CSAT ≥ 4.5 作为证据。

### 约束与边缘情况
- **反幻觉硬约束**: 查不到就转人工,禁止编造
- **合规硬约束**: GDPR / CCPA / 个保法检查不阻塞沙箱,但阻塞对外开放
- **人工离线**: 超过 180s 未接单,卡片退回 Agent 并发安抚消息
- **并发上限**: 一个客服同时 Copilot ≤ 5 个对话(配置可调)
- **回滚**: `/rollback <proposal_id>` 立即撤回灰度中的梦境提案

## 8. Out of Scope(本版本不做)

- **语音合成/识别的深度优化** —— PSTN 接入做基础,高级声纹/情感合成放后续
- **定制模型训练** —— 只做 RAG 和 prompt 工程,不微调
- **深度 CRM 双向写入** —— 本期只做单向推送线索,CRM 同步放后续
- **白标** —— 本期所有租户共享 AutoService 品牌,白标作为企业版再谈
- **移动端 App** —— 客服工作区本期只做 Web + IM 插件

## 9. Dependencies & Risks

### 依赖
- **基础设施**: 快慢双模型推理集群 · 多租户沙箱 · 灰度发布通道
- **外部**: 20+ 语种 LLM 可用性 · 独立站嵌入 SDK · IM 渠道 SDK(微信/WhatsApp/Slack 等)· PSTN 网关
- **内部 OneSync**: zchat v0.3.x 作为 IM 底层 · Socialware Role/Scope/Commitment/Flow 框架 · hackforger 设计系统 "江峡泼墨"

### 风险与缓解
| 风险 | 影响 | 缓解 |
|---|---|---|
| 反幻觉机制漏洞导致客户损失 | 高 | 双模型 cross-check + 知识库强约束 + 人工灰度 |
| 商户对"自助上线"期望过高 | 中 | 首屏明确标注"沙箱先行",审核并行不阻塞 |
| 闲时学习提案质量差 | 中 | 5% 灰度起步 + 商户审核 + `/rollback` |
| 多租户数据串漏 | 致命 | 严格沙箱隔离 + 每租户独立知识库 |
| SLA 达不成稳态 | 高 | 陪跑阶段承诺,不承诺首月达标 |

## 10. Open Questions

1. **计费模式**: 按接管次数阶梯 vs 按月订阅 + 超额 —— 定价团队 before Q3
2. **Copilot 提议采纳率**是否也纳入计费选项?(当前只是监控指标)

6. 20+ 种语言的**合规地区映射**是否需要按语言而非按商户地区自动切换?
