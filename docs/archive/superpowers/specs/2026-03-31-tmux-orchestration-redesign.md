# Tmux Orchestration Redesign

**Date:** 2026-03-31
**Status:** Draft

## Problem

Tmux orchestration in zchat has accumulated reliability issues rooted in timing-based hacks:

1. **Blind sleep for startup confirmation** — `agent_manager._spawn_tmux()` uses `sleep 3 && tmux send-keys Enter` to auto-confirm Claude Code's two startup prompts (folder trust + development channel). This fails when Claude takes longer to start, causing the E2E `test_second_agent` to be permanently flaky.

2. **No readiness detection** — After creating a pane, `agent create` returns immediately. There is no mechanism to know when Claude has fully started. E2E tests compensate with 30-second timeouts.

3. **Imperative session layout** — Session creation is scattered across `agent_manager.py` (agent panes) and `irc_manager.py` (WeeChat pane). The layout isn't declarative or reproducible — after `shutdown`, you cannot restore the previous session.

4. **Cramped pane layout** — All components (WeeChat + N agents) share a single tmux window as panes, each getting a fraction of terminal space.

5. **Ephemeral agent workspaces** — Agent workspaces are in `/tmp/`, lost on reboot. Agent state (`.claude/`, `.mcp.json`) cannot survive restarts.

## Solution

Three coordinated changes:

1. **tmuxp for declarative session layout** — session structure defined in YAML, reproducible via `tmuxp load`
2. **SessionStart hook for readiness detection** — Claude Code's hook system writes a ready marker; agent_manager polls for it
3. **Persistent agent workspaces + windows** — agents get dedicated tmux windows and persistent directories under the project

## Part 1: Directory Structure

### Current

```
~/.zchat/projects/<name>/
├── config.toml
└── state.json
```

Agent workspaces: `/tmp/zchat-{scoped_name}/` (ephemeral)

### New

```
~/.zchat/projects/<name>/
├── config.toml
├── state.json
├── tmuxp.yaml              # declarative session layout
├── bootstrap.sh            # before_script for tmuxp
└── agents/
    ├── <scoped_name>/      # persistent agent workspace
    │   ├── .claude/
    │   │   └── settings.local.json  (permissions, plugins, hooks)
    │   ├── .mcp.json
    │   ├── .zchat-env
    │   └── .claude-plugin -> (symlink to channel server plugin)
    ├── <scoped_name>.ready # ready marker (touched by SessionStart hook)
    └── ...
```

### Changes

**`agent_manager._create_workspace()`:** Change from `tempfile.gettempdir()` to `~/.zchat/projects/<name>/agents/<scoped_name>/`. The workspace persists across restarts.

**`agent_manager._cleanup_workspace()`:** On `agent stop`, delete only the ready marker, NOT the workspace directory. The workspace is reused on `agent restart`.

## Part 2: tmuxp Declarative Session Layout

### Dependencies

Add `tmuxp` to `pyproject.toml` dependencies (alongside existing `libtmux`).

### tmuxp.yaml

Generated by `project create`. Updated by `irc start` (adds WeeChat window). Agent windows are NOT synced to YAML during dynamic operations — they are reconstructed from `state.json` on session restore.

**Initial (after `project create`):**
```yaml
session_name: zchat-<uuid>-<name>
start_directory: /absolute/path/to/.zchat/projects/<name>
before_script: /absolute/path/to/.zchat/projects/<name>/bootstrap.sh
windows:
  - window_name: main
    panes:
      - blank
```

**After `zchat irc start`:**
```yaml
session_name: zchat-<uuid>-<name>
start_directory: /absolute/path/to/.zchat/projects/<name>
before_script: /absolute/path/to/.zchat/projects/<name>/bootstrap.sh
windows:
  - window_name: weechat
    panes:
      - shell_command:
          - weechat --dir /absolute/path/to/.zchat/projects/<name>/weechat ...
    focus: true
```

Note: all paths in YAML use absolute expanded paths (no `~` — tilde is not expanded inside YAML strings).

### bootstrap.sh

Generated alongside `tmuxp.yaml`. Runs before session creation via tmuxp's `before_script`. Uses absolute paths (generated by Python with full expansion):

```bash
#!/bin/bash
set -euo pipefail
PROJECT_DIR="/absolute/path/to/.zchat/projects/<name>"
mkdir -p "$PROJECT_DIR/agents"
# Only clean ready markers for agents not listed as running in state.json
# (avoids race if session is restored while agents are still alive)
for f in "$PROJECT_DIR/agents"/*.ready; do
    [ -f "$f" ] || continue
    agent=$(basename "$f" .ready)
    if ! grep -q "\"$agent\".*\"running\"" "$PROJECT_DIR/state.json" 2>/dev/null; then
        rm -f "$f"
    fi
done
```

### Session Restoration

On `zchat irc start`, if agents exist in `state.json` with status "running" from a previous session, the `irc start` flow:
1. `tmuxp load -d tmuxp.yaml` creates the session (detached, avoids interactive prompt)
2. Reads `state.json` for previously running agents
3. For each agent: calls `session.new_window()` to recreate the agent window
4. Attaches to the session

This avoids the dual-source-of-truth problem — `state.json` is the single source for agent state, YAML only defines the base session layout (WeeChat).

### Operation Mapping

| CLI Command | tmuxp | libtmux | State Update |
|------------|-------|---------|-------------|
| `project create` | Generate YAML + bootstrap.sh | — | — |
| `zchat irc start` | `tmuxp load -d` | Recreate agent windows from state.json | — |
| `zchat agent create` | — | `session.new_window(window_name=name, window_shell=cmd, attach=False)` | Add to state.json |
| `zchat agent stop` | — | kill window | Update state.json |
| `zchat shutdown` | — | kill session | Mark all agents offline in state.json |
| `zchat irc start` (再次) | `tmuxp load -d` | Recreate agent windows from state.json | — |

### Key Design Decisions

- **`tmuxp load -d` on `irc start`** — creates the session in detached mode (avoids "Already inside TMUX" interactive prompt), then attaches
- **YAML is base layout only** — defines session name, WeeChat window, bootstrap. Agent windows are NOT stored in YAML
- **`state.json` is single source of truth for agents** — agent windows are reconstructed from state on session restore
- **Each component is a window** — WeeChat and each agent get their own window (not pane), providing full terminal space

### tmuxp Plugin (optional, future)

tmuxp supports plugins via `TmuxpPlugin` with hooks like `on_window_create`. A `ZchatPlugin` could inject readiness monitoring on window creation. This is a future enhancement — the initial implementation uses libtmux directly.

## Part 3: Readiness Detection via SessionStart Hook

### Problem

Claude Code shows two confirmation prompts on startup:
1. "Yes, I trust this folder" — workspace trust
2. "I am using this for local development" — development channel confirmation

These must be auto-confirmed via `tmux send-keys`, but the current `sleep 3` approach is unreliable.

After both confirmations, Claude shows its version banner (e.g., `Claude Code v2.1.88`), and the `SessionStart` hook fires.

### Solution: Two-Phase Startup

**Phase 1: Confirmation polling (capture-pane)**

After creating the agent window, start a daemon `threading.Thread` that:
1. Polls `tmux capture-pane` every 0.5s via libtmux's `capture_pane()` (returns `list[str]`)
2. Detects prompt text (`"I trust this folder"` or `"local development"`)
3. Sends Enter via libtmux `pane.send_keys("", enter=True)` when detected
4. Continues polling until the ready marker appears (Phase 2 signal)

Note: The prompt text patterns are Claude Code UI strings that may change between versions. As a fallback, also match broader patterns (e.g., lines containing `"Enter to confirm"`). If `--permission-mode bypassPermissions` starts skipping prompts in future Claude versions, the polling simply times out harmlessly (Phase 2 still works).

**Phase 2: Ready marker (SessionStart hook)**

The agent's `.claude/settings.local.json` includes a `SessionStart` hook. The hook command uses **literal absolute paths** (expanded at write time by `start.sh`), not environment variables at hook execution time — this avoids reliance on Claude Code's hook executor having the right env context:

```json
{
  "hooks": {
    "SessionStart": [{
      "matcher": "startup",
      "hooks": [{
        "type": "command",
        "command": "touch /Users/x/.zchat/projects/local/agents/alice-agent0.ready"
      }]
    }]
  },
  "permissions": {
    "allow": [
      "mcp__zchat-channel__reply",
      "mcp__zchat-channel__join_channel"
    ]
  },
  "enabledPlugins": {
    "zchat@ezagent42": true
  }
}
```

**Note on hook schema:** The hook format above must be verified against Claude Code's actual schema before implementation. Run a minimal test: create a `settings.local.json` with a `SessionStart` hook and confirm it fires.

### Path Expansion in start.sh

`start.sh` builds `settings.local.json` using `jq` (consistent with how `.mcp.json` is already built). The `ZCHAT_PROJECT_DIR` and `AGENT_NAME` environment variables are available because `start.sh` sources `.zchat-env`:

```bash
# In start.sh, after sourcing .zchat-env:
READY_PATH="${ZCHAT_PROJECT_DIR}/agents/${AGENT_NAME}.ready"

jq -n \
  --arg ready_cmd "touch $READY_PATH" \
  '{
    hooks: {
      SessionStart: [{
        matcher: "startup",
        hooks: [{ type: "command", command: $ready_cmd }]
      }]
    },
    permissions: { allow: [...] },
    enabledPlugins: { "zchat@ezagent42": true }
  }' > .claude/settings.local.json
```

This produces a JSON file with literal expanded paths — no env var references in the final JSON.

### Environment Variable Chain

1. `agent_manager` knows the project dir → writes to `.zchat-env`:
   ```
   export ZCHAT_PROJECT_DIR=/Users/x/.zchat/projects/local
   ```
2. `start.sh` sources `.zchat-env` → `ZCHAT_PROJECT_DIR` is in the shell environment
3. `start.sh` uses `jq` to write `.claude/settings.local.json` with **literal expanded path** (e.g., `touch /Users/x/.zchat/projects/local/agents/alice-agent0.ready`)
4. Claude Code starts → `SessionStart` hook fires → executes the literal `touch` command

### Agent Startup Timeline

```
t=0    agent_manager creates window (libtmux session.new_window)
t=0    background confirmation thread starts polling capture-pane
t=~2s  Claude shows "I trust this folder" → thread sends Enter
t=~4s  Claude shows "local development" → thread sends Enter
t=~8s  Claude fully starts → SessionStart hook fires → writes .ready marker
t=~8s  agent_manager detects .ready → marks agent "running"
```

### Timeout and Error Handling

- Confirmation polling timeout: 60 seconds → if no prompts detected, log warning
- Ready marker timeout: 60 seconds → if .ready not created, mark agent as "error"
- On `agent stop`: delete `.agents/<name>.ready`
- On `bootstrap.sh` (session start): clean all stale `.ready` files

### Agent Stop Flow

When stopping an agent:
1. Delete `.agents/<name>.ready` marker
2. Find the window's first pane, send pre_stop hook (`/exit`) via `pane.send_keys`
3. Poll for window death (up to 10s)
4. If still alive, kill window via libtmux
5. Update `state.json` (mark offline)
6. Workspace directory is preserved (for restart)

## Part 4: Files Changed

### New Files

| File | Purpose |
|------|---------|
| `~/.zchat/projects/<name>/tmuxp.yaml` | Declarative session layout |
| `~/.zchat/projects/<name>/bootstrap.sh` | Pre-session bootstrap script |
| `~/.zchat/projects/<name>/agents/` | Agent workspaces + ready markers |

### Modified Files

**`zchat/cli/agent_manager.py`:**
- `_create_workspace()`: use `projects/<name>/agents/<scoped_name>/` instead of `/tmp/`
- `_spawn_tmux()`: replace `window.split()` with `session.new_window()`; replace `sleep 3 + Enter` Popen with confirmation polling thread + ready marker wait
- `_cleanup_workspace()`: only delete ready marker on stop, preserve workspace
- `_force_stop()`: kill window instead of kill pane
- `_check_alive()`: check window existence instead of pane
- New: `_build_env_context()` adds `ZCHAT_PROJECT_DIR`
- New: `_wait_for_ready()` polls `.agents/<name>.ready` with timeout
- New: `_auto_confirm_startup()` polls capture-pane for prompts, sends Enter

**`zchat/cli/irc_manager.py`:**
- `start_weechat()`: replace `window.split()` with YAML update + `tmuxp load` (or `session.new_window()` if session already exists)
- `stop_weechat()`: kill window instead of kill pane

**`zchat/cli/project.py`:**
- `create_project_config()`: also generate `tmuxp.yaml` and `bootstrap.sh`

**`zchat/cli/tmux.py`:**
- Add `find_window(session, window_name)` helper
- Add `window_alive(session, window_name)` helper
- Deprecate pane-focused helpers (keep for backward compat)

**`zchat/cli/templates/claude/start.sh`:**
- Write `SessionStart` hook in `.claude/settings.local.json`
- Read `ZCHAT_PROJECT_DIR` from environment for hook command

**`pyproject.toml`:**
- Add `tmuxp` dependency

### State File Changes

**`state.json`:** Agent records change from `pane_id` to `window_name`:

```json
{
  "agents": {
    "alice-agent0": {
      "type": "claude",
      "workspace": "~/.zchat/projects/local/agents/alice-agent0",
      "window_name": "alice-agent0",
      "status": "running",
      "created_at": 1711843200,
      "channels": ["#general"]
    }
  }
}
```

### State File Migration

Existing `state.json` files use `pane_id` fields. On load, if an agent record contains `pane_id` but no `window_name`, treat it as offline. No explicit migration step needed — the agent will be in "offline" state and can be re-created.

### Shutdown Behavior

`zchat shutdown` kills the tmux session (behavior change — currently it only stops agents/WeeChat/ergo individually). This is cleaner: all windows die with the session. The YAML and `state.json` are preserved for the next `zchat irc start`.
